Yes—this is doable to fix/work around, and your suspicion is basically correct: ClickHouse is mistakenly trying to resolve a literal hostname like secure://100.65.0.116, which obviously is not a valid DNS name.

What’s actually happening

system.zookeeper_info uses a low-level ZooKeeper “four-letter word” style connection. In some ClickHouse versions (including the 26.2–26.3 range you’re on), when <secure>1</secure> is used in the ZK/Keeper config, the code path incorrectly prefixes the endpoint with a scheme like:

secure://100.65.0.116

Then it passes that string into Poco’s DNS resolver, which treats it as a hostname → hence:

Host not found: secure://100.65.0.116

So it’s not actually TLS failing; it’s name resolution happening on a malformed host string.

Why it shows up only in system.zookeeper_info

Other parts of replication / Keeper work because they use the proper Keeper client with TLS handling.

But system.zookeeper_info is special: it probes servers directly using a simplified connection mechanism that doesn’t fully respect the same TLS abstraction layer.

Workarounds (practical options)

1) Upgrade ClickHouse (best fix)

This is the cleanest solution. This class of “secure:// inserted into host” bugs has been fixed in later patches around the 26.x line.

👉 If you can, move to the latest 26.3.x or 26.4+ patch level.

2) Avoid <secure>1</secure> in ZK config

If your Keeper is already behind TLS at the transport level, try removing:

XML
<secure>1</secure>

and instead rely on explicit SSL settings in the client/server sections.

This forces ClickHouse to treat it as a plain TCP endpoint while TLS is handled elsewhere.

3) Use hostnames instead of raw IPs

Oddly, some internal code paths behave differently when resolving IP literals vs names.

Try:

XML
<host>keeper-1.internal</host>

instead of 100.65.x.x

This is not ideal, but it can avoid the malformed URI parsing path.

4) Disable system.zookeeper_info usage (last resort)

If everything else works and you only need observability:

  • avoid querying system.zookeeper_info

  • rely on Keeper logs / mntr / srvr via external tooling

Bottom line

Yes, it’s fixable—but it’s not a config issue in your cluster logic. It’s a ClickHouse-side bug/edge case in how secure Keeper endpoints are passed into the system table query path.

If you want, paste your full Keeper + ClickHouse version matrix (all nodes), and I can tell you the safest upgrade or exact patch version where this is fixed.
